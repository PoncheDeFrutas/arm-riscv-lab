16 · ELF, linking, loading y binarios

Entiende el binario como objeto técnico: fuente, objeto, ejecutable, segmentos, loader y proceso.

Página principal de la Unidad 16 de la ruta AArch64.

Cursos > AArch64 > ELF, linking, loading y binarios

Hasta ahora escribiste .s, generaste .o, enlazaste ejecutables, llamaste syscalls, escribiste funciones y mezclaste C con assembly. Ahora vamos a abrir el binario. No como curiosidad, sino como parte del contrato técnico que une assembler, linker, loader, memoria virtual y proceso Linux.

Qué debe quedar claro

Esta unidad no trata de memorizar readelf, objdump y nm. Trata de responder una pregunta más importante:

¿qué cambió entre mi archivo fuente y el proceso que Linux ejecuta?
Etapa Archivo Herramienta Qué contiene
fuente .s editor assembly legible
objeto .o assembler código máquina, símbolos, relocations
ejecutable ELF linker segmentos, entry point, símbolos resueltos o dinámica
proceso memoria loader/kernel segmentos mapeados con permisos
NotaIdea clave

El archivo ejecutable no es una caja negra. Tiene estructura: header, secciones, segmentos, símbolos, direcciones, permisos y, a veces, dependencias dinámicas.

Esta estructura existe porque distintas herramientas necesitan mirar el mismo archivo con preguntas diferentes. El assembler piensa en secciones y símbolos. El linker piensa en combinar piezas, resolver nombres y producir un layout final. El loader piensa en rangos de archivo que se convertirán en regiones de memoria. El depurador y las herramientas de inspección vuelven a usar símbolos, secciones y direcciones para ayudarte a explicar qué pasó.

Por eso esta unidad insiste en separar capas. Si mezclas todas las palabras, el tema se vuelve una lista de comandos. Si separas capas, cada comando tiene una función clara: readelf -h identifica, readelf -S organiza, readelf -l explica carga, objdump -d muestra instrucciones, nm muestra nombres y strip enseña qué información era necesaria para ejecutar y cuál era útil para inspeccionar.

Mapa rápido

1. Produce .s -> .o -> ELF.

2. Identifica ELF header y entry point.

3. Organiza Secciones, símbolos y relocations.

4. Mapea Program headers, segmentos y permisos.

5. Carga Loader, dynamic linking, GOT/PLT y PIE.

Flujo completo

Cada flecha resuelve un problema distinto. El assembler traduce líneas y crea relocations cuando todavía faltan direcciones. El linker combina objetos, resuelve símbolos y produce un archivo ejecutable. El loader usa program headers para crear el proceso en memoria.

La flecha más importante para esta unidad es la que va de ELF executable a proceso. Un archivo en disco no ejecuta nada por sí mismo. Linux debe leer su estructura, mapear partes del archivo en memoria, asignar permisos, preparar el estado inicial y transferir control al entry point. Ese paso explica por qué no basta con saber escribir instrucciones A64: también debes saber cómo esas instrucciones quedan empaquetadas y cargadas.

Sección no es segmento

Concepto Uso principal Lo ve principalmente
sección organizar contenido para assembler/linker linker, herramientas
segmento mapear partes del ejecutable en memoria loader/kernel

La sección ayuda a construir y analizar. El segmento ayuda a ejecutar. Por eso .text, .rodata, .data y .bss pueden terminar agrupadas en menos segmentos con permisos como R, RX o RW.

Herramientas de la unidad

Herramienta Uso inicial
readelf -h ver ELF header y entry point
readelf -S ver section headers
readelf -l ver program headers / segmentos
objdump -d desensamblar código
nm ver símbolos
ldd ver dependencias dinámicas
strip remover símbolos no necesarios para ejecución

Usa el prefijo aarch64-linux-gnu- cuando trabajes en host x86_64 con toolchain cruzado. En ARM64 nativo, las herramientas sin prefijo suelen bastar.

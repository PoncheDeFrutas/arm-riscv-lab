17 · Debugging con GDB, QEMU, strace y lectura de memoria

Leer el estado real del programa mientras se ejecuta.

Página principal de la Unidad 17 de la ruta AArch64.

Cursos > AArch64 > Debugging con GDB, QEMU y strace

Hasta ahora escribiste programas, entendiste registros, memoria, stack, heap, mmap, ABI y ELF. Esta unidad cambia el punto de vista: ya no basta con saber qué debería hacer el programa. Ahora debes observar qué está haciendo realmente mientras corre.

Objetivo de la unidad

El objetivo central es:

leer el estado real del programa mientras se ejecuta

Debugging no es memorizar comandos de GDB. Es construir una forma de mirar: detener el programa, observar registros, avanzar una instrucción, comprobar una dirección, mirar bytes de memoria, verificar syscalls y reconstruir por qué el estado final no coincide con lo que esperabas.

Esta unidad une varias piezas previas:

Ya viste… Aquí lo observarás con…
Unidad 03: registros, PC, SP, x29, x30, PSTATE info registers, layout regs
Unidad 06: dirección vs contenido, .data, .bss x/x, x/s, símbolos
Unidad 09 y 10: syscalls y errores strace
Unidad 11: stack frames sp, x29, x30, bt
Unidad 13: mmap, permisos y regiones inspección de direcciones mapeadas
Unidad 16: ELF, símbolos, entry point breakpoints, desensamblado y símbolos

Mapa mental

La idea no es usar todas las herramientas a la vez. Cada herramienta responde una pregunta distinta. GDB responde “qué estado tiene el proceso ahora”. QEMU responde “cómo ejecuto AArch64 cuando mi host no es ARM64”. strace responde “qué le pidió el programa al kernel”. Un core dump responde “qué estado quedó guardado después de fallar”.

Herramientas de la unidad

Herramienta Uso principal en esta unidad
gdb-multiarch depurar AArch64 desde host no ARM
gdb depurar nativamente en ARM64
qemu-aarch64 ejecutar binarios AArch64 en host x86_64
qemu-aarch64 -g esperar conexión remota de GDB
strace observar syscalls y errores

En el laboratorio cruzado usamos gdb-multiarch con qemu-aarch64 -g. En un equipo ARM64 nativo, el flujo puede ser más directo con gdb ./programa.

Comandos GDB iniciales

Comando Uso inicial
break etiqueta detener en una etiqueta o función
run iniciar el programa en una sesión local
continue continuar hasta el siguiente breakpoint o fin
stepi ejecutar una instrucción entrando en llamadas
nexti ejecutar una instrucción sin entrar en llamadas
info registers ver registros
x/i $pc ver instrucción actual
x/x dirección ver memoria en hexadecimal
x/s dirección ver string en memoria
disassemble ver instrucciones de una función
layout regs vista TUI de registros
layout asm vista TUI de ensamblador

La tabla no es una lista para memorizar. Es una caja de herramientas. Primero detienes, luego miras el estado, después avanzas y comparas qué cambió.

Memoria inspeccionable

Región Qué observar
.data datos inicializados modificables
.bss buffers reservados sin inicialización explícita
heap bloques dinámicos, punteros y contenido
mmap regiones mapeadas, permisos y accesos
stack frames, retorno, locales temporales

La pregunta siempre debe ser concreta: “¿esta dirección es el puntero o el contenido?”, “¿este registro cambió antes o después de la instrucción?”, “¿esta llamada al kernel devolvió éxito o error?”.

Qué no estudiaremos todavía

Tema Por qué queda fuera

Python API de GDB automatización viene después
DWARF completo solo usaremos símbolos básicos
reverse engineering avanzado aquí depuramos programas conocidos
exploit development, ROP, heap exploitation seguridad ofensiva requiere otra unidad
debugging de kernel o bare metal completo este bloque es Linux userland
perf, sanitizers, rr, eBPF herramientas avanzadas para unidades posteriores

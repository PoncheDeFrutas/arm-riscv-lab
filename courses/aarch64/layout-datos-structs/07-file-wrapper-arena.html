14.7 · File wrapper y arena simple

Descriptores con recursos: fd del kernel y bloques de memoria propios.

File wrapper, arena simple, ownership, cleanup y patrones de recursos.

Cursos > AArch64 > Layout de datos > File wrapper y arena

Algunos objetos no solo guardan datos. También poseen recursos: fd abiertos, regiones mapeadas o bloques de memoria. Eso exige constructor, destructor y cleanup claro.

File wrapper

Un fd viene del kernel, pero tu programa puede envolverlo en un descriptor propio:

.equ FILE_FD,    0
.equ FILE_FLAGS, 8
.equ FILE_SIZE,  16

Tabla:

Campo Offset Tamaño Uso
fd 0 8 fd retornado por openat
flags 8 8 estado propio del wrapper
FILE_SIZE 16 - tamaño total

Invariante mínima:

fd >= 0 si archivo está abierto
fd = -1 si wrapper no posee archivo

Constructor de file wrapper

Concepto:

file_init
  fd = -1
  flags = 0

Código:

// x0 = self
file_init:
    mov x1, #-1
    str x1, [x0, #FILE_FD]
    str xzr, [x0, #FILE_FLAGS]
    ret

Inicializar fd = -1 evita cerrar basura si destructor corre antes de abrir.

Estado abierto y cerrado

Un wrapper debe poder decir si posee recurso:

fd = -1
  no hay archivo abierto

fd >= 0
  hay archivo abierto
  destructor debe cerrar

Eso evita dos errores: cerrar basura y cerrar dos veces. El campo fd no solo guarda número; también participa en estado del objeto.

Destructor de file wrapper

Concepto:

si fd >= 0
  close(fd)
  fd = -1

Código esquemático:

// x0 = self
file_close:
    ldr x1, [x0, #FILE_FD]
    cmp x1, #0
    b.lt file_close_done

    mov x2, x0
    mov x0, x1
    mov x8, #57
    svc #0

    mov x1, #-1
    str x1, [x2, #FILE_FD]

file_close_done:
    ret

Se guarda self en x2 porque syscall usa x0 para argumento y retorno.

Después de close, el wrapper marca fd = -1. Esa escritura importa tanto como la syscall. Sin ella, el descriptor propio dice “sigo abierto” aunque el kernel ya cerró el recurso.

Programa completo: File wrapper

Este programa crea /tmp/u14_file_wrapper_demo.txt, escribe un mensaje y cierra el fd mediante file_close.

Guarda como file-wrapper.s.

.equ FILE_FD,    0
.equ FILE_FLAGS, 8
.equ FILE_SIZE,  16

.equ AT_FDCWD, -100
.equ O_FLAGS, 577        // O_WRONLY | O_CREAT | O_TRUNC

.data
file_obj:
    .quad -1
    .quad 0

filename:
    .asciz "/tmp/u14_file_wrapper_demo.txt"

message:
    .ascii "archivo creado desde file wrapper\n"
message_len = . - message

.text
.global _start
_start:
    ldr x0, =file_obj
    bl file_init

    mov x0, #AT_FDCWD
    ldr x1, =filename
    mov x2, #O_FLAGS
    mov x3, #0644
    mov x8, #56
    svc #0
    cmp x0, #0
    b.lt error

    ldr x9, =file_obj
    str x0, [x9, #FILE_FD]

    mov x10, x0
    mov x0, x10
    ldr x1, =message
    mov x2, #message_len
    mov x8, #64
    svc #0
    cmp x0, #0
    b.lt cleanup_error

    ldr x0, =file_obj
    bl file_close

    mov x0, #0
    mov x8, #93
    svc #0

cleanup_error:
    ldr x0, =file_obj
    bl file_close

error:
    mov x0, #1
    mov x8, #93
    svc #0

file_init:
    mov x1, #-1
    str x1, [x0, #FILE_FD]
    str xzr, [x0, #FILE_FLAGS]
    ret

file_close:
    ldr x1, [x0, #FILE_FD]
    cmp x1, #0
    b.lt file_close_done

    mov x3, x0
    mov x0, x1
    mov x8, #57
    svc #0

    mov x1, #-1
    str x1, [x3, #FILE_FD]

file_close_done:
    ret

Construir:

$ aarch64-linux-gnu-as file-wrapper.s -o file-wrapper.o
$ aarch64-linux-gnu-ld file-wrapper.o -o file-wrapper
$ qemu-aarch64 ./file-wrapper
$ echo $?
0

Ruta práctica:

$ cd examples/aarch64/14-layout-datos-structs/04-file-wrapper/src
$ make
$ make run
$ echo $?
0

Arena simple

Arena es un bloque grande del que se entregan pedazos en orden. No libera cada pedazo por separado; se reinicia o libera toda arena.

.equ ARENA_BASE, 0
.equ ARENA_USED, 8
.equ ARENA_CAP,  16
.equ ARENA_SIZE, 24

Diagrama:

base -> [ usados ][ libre................ ]
         0       used                    cap

Invariante:

0 <= used <= cap
base apunta a bloque de cap bytes

Arena como descriptor + bloque

Arena también separa metadata y memoria real:

arena descriptor
  base -> bloque grande
  used -> bytes entregados
  cap  -> tamaño total

bloque grande
  objetos temporales uno tras otro

arena_alloc no pide memoria al kernel en cada llamada. Solo mueve used.

Operación arena_alloc

Idea:

si used + n <= cap
  ptr = base + used
  used = used + n
  retornar ptr
si no
  retornar error

Código conceptual:

// x0 = self Arena*
// x1 = n bytes
arena_alloc:
    ldr x2, [x0, #ARENA_BASE]
    ldr x3, [x0, #ARENA_USED]
    ldr x4, [x0, #ARENA_CAP]

    add x5, x3, x1
    cmp x5, x4
    b.hi arena_full

    add x6, x2, x3
    str x5, [x0, #ARENA_USED]
    mov x0, x6
    ret

arena_full:
    mov x0, #0
    ret

Aquí 0 representa fallo conceptual. En programa real, elegirías contrato explícito de error.

Alineación dentro de arena

Si entregas objetos de 8 bytes, conviene que used avance con alineación. En esta unidad no implementamos función completa de alineación, pero la regla mental queda:

pedir n bytes
redondear n si hace falta
mantener used alineado

Si ignoras esto, un objeto dentro de arena puede empezar en dirección incómoda para sus campos.

Por qué arena ayuda

Arena simplifica muchos objetos pequeños con misma vida útil:

crear arena
crear varios objetos dentro
usar objetos
resetear o liberar arena completa

No reemplaza malloc. Es patrón simple para entender ownership grupal.

Reset de arena

Reset conceptual:

arena_reset
  used = 0

Eso no borra bytes. Solo dice que todos los pedazos entregados antes dejan de ser válidos. Es parecido a buffer_clear: cambia estado lógico, no limpia cada byte.

Después de reset, usar punteros viejos dentro de la arena es error conceptual.

Programa completo: Arena simple

Este programa reserva dos bloques de 4 bytes desde la arena, escribe ABCD y EFGH, luego hace reset lógico (used = 0).

Guarda como arena-simple.s.

.equ ARENA_BASE, 0
.equ ARENA_USED, 8
.equ ARENA_CAP,  16
.equ ARENA_SIZE, 24

.bss
arena_storage:
    .skip 16

.data
arena_desc:
    .quad arena_storage
    .quad 0
    .quad 16

.text
.global _start
_start:
    ldr x0, =arena_desc
    mov x1, #4
    bl arena_alloc
    cbz x0, error

    mov w2, #'A'
    strb w2, [x0]
    mov w2, #'B'
    strb w2, [x0, #1]
    mov w2, #'C'
    strb w2, [x0, #2]
    mov w2, #'D'
    strb w2, [x0, #3]

    ldr x0, =arena_desc
    mov x1, #4
    bl arena_alloc
    cbz x0, error

    mov w2, #'E'
    strb w2, [x0]
    mov w2, #'F'
    strb w2, [x0, #1]
    mov w2, #'G'
    strb w2, [x0, #2]
    mov w2, #'H'
    strb w2, [x0, #3]

    ldr x0, =arena_desc
    bl arena_reset

    mov x0, #0
    mov x8, #93
    svc #0

error:
    mov x0, #1
    mov x8, #93
    svc #0

arena_alloc:
    ldr x2, [x0, #ARENA_BASE]
    ldr x3, [x0, #ARENA_USED]
    ldr x4, [x0, #ARENA_CAP]

    add x5, x3, x1
    cmp x5, x4
    b.hi arena_full

    add x6, x2, x3
    str x5, [x0, #ARENA_USED]
    mov x0, x6
    ret

arena_full:
    mov x0, #0
    ret

arena_reset:
    str xzr, [x0, #ARENA_USED]
    ret

$ aarch64-linux-gnu-ld arena-simple.o -o arena-simple
$ qemu-aarch64 ./arena-simple

Errores comunes

Error Consecuencia
cerrar fd inválido error de syscall
no marcar fd como cerrado double close futuro
arena sin revisar capacidad overflow
usar puntero de arena después de reset use-after-free conceptual
liberar pedazos individuales de arena rompe modelo simple
confundir arena con allocator general expectativas falsas

Mini ejercicios

  1. ¿Por qué fd = -1 representa “sin archivo”?
  2. ¿Por qué file_close guarda self antes de syscall?
  3. ¿Qué campos necesita una arena?
  4. ¿Qué invariante protege arena_alloc?
  5. ¿Por qué arena no libera objetos individuales?

Cierre

File wrapper y arena muestran que layout manual no es solo para datos bonitos. También organiza recursos, ownership y cleanup.

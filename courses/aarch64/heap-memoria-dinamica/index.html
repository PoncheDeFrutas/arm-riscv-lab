12 · Heap y memoria dinámica

Vida útil, ownership, bloques dinámicos y errores de memoria.

Página principal de la Unidad 12 de la ruta AArch64.

Cursos > AArch64 > Heap y memoria dinámica

Ya viste memoria, punteros, stack frames y recursos del kernel. Ahora toca una pregunta central: ¿qué pasa cuando un dato debe vivir más que una función, o cuando su tamaño se decide mientras el programa corre? Para eso existe el heap.

Mapa rápido

1. Ubica .data, .bss, stack y heap tienen vidas distintas.

2. Reserva Un bloque dinámico tiene dirección y capacidad.

3. Usa No todo lo reservado está usado.

4. Posee Ownership dice quién debe liberar.

5. Libera Después de liberar, puntero viejo no debe usarse.

Regiones de memoria

Región Vida útil típica Tamaño Uso
.data / .bss todo el proceso fijo al cargar datos globales o estáticos
stack llamada actual y frames activos cambia con llamadas locales, retorno, frames
heap hasta que se libere dinámico buffers, strings, estructuras que crecen 
Direcciones bajas

.text
.rodata
.data

Direcciones altas

El orden real depende de sistema y configuración. El mapa sirve para estudiar clases de memoria y vida útil.

Qué problema resuelve cada región

Pregunta Respuesta típica
¿El dato es fijo y existe durante todo el proceso? .rodata, .data o .bss
¿El dato pertenece a una llamada activa? stack
¿El dato debe vivir después de retornar? heap
¿El tamaño se conoce al escribir el .s? .bss puede alcanzar
¿El tamaño depende de entrada o crecimiento? heap

El heap no existe porque el stack sea “malo”. Existe porque no todos los datos tienen la misma vida. Una variable local vive mientras su frame existe. Un buffer dinámico puede necesitar sobrevivir a varias funciones o cambiar de tamaño.

Idea central

Un bloque dinámico se entiende como:

bloque
  dirección: puntero guardado en registro o memoria
  capacidad: bytes reservados
  usados: bytes con datos válidos
  dueño: parte del programa responsable de liberar
NotaIdea clave

Reservar memoria no significa usarla toda. Un buffer puede tener capacidad de 64 bytes y solo 12 bytes válidos.

Glosario mínimo

Palabra Lectura práctica
bloque rango continuo de bytes reservado para un uso
puntero dirección inicial del bloque
capacidad cantidad total de bytes reservados
usados cantidad de bytes que contienen datos válidos
dueño parte del programa responsable de liberar
liberado bloque que ya no debe usarse

Este vocabulario aparecerá todo el tiempo. Si solo recuerdas “dirección”, te faltará media historia. El programa también necesita saber cuánto cabe, cuánto se usó y quién es responsable.

Ciclo completo

reservar bloque
  guardar puntero
  guardar capacidad

usar bloque
  revisar límites
  actualizar bytes usados

liberar bloque
  dejar de usar puntero viejo
  evitar liberar otra vez

La Unidad 10 enseñó recursos como file descriptors. Memoria dinámica también es un recurso, pero no se libera con close: se libera con el mecanismo de memoria correspondiente.

Errores que esta unidad prepara

Error Qué significa Consecuencia
memory leak se pierde referencia a memoria reservada memoria no se puede liberar
use-after-free se usa un puntero después de liberar comportamiento incorrecto o inseguro
double free se libera dos veces el mismo bloque corrupción o error del allocator

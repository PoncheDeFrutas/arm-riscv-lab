03 · Modelo ARMv8-A / AArch64

Registros generales, registros especiales, flags, SIMD/Floating Point y niveles de excepción como estado visible del procesador.

Página principal de la Unidad 03 de la ruta AArch64.

Cursos > AArch64 > Modelo ARMv8-A / AArch64

Esta unidad muestra qué estado puede observar un programa AArch64: registros, flags, direcciones de ejecución y parte del contexto que aparece al depurar. No buscamos memorizar el manual, sino saber qué significa mirar x0, sp, pc, x30 o NZCV durante un programa real.

TipMaterial de apoyo para clase

Esta presentación resume el mapa de la unidad para clase. Usa esta página como referencia completa y el deck como apoyo visual.

Para iniciar la presentación, abre el enlace y avanza con las flechas del teclado o la barra espaciadora.

Ver presentación Slidev

Mapa rápido

1. Generales X0-X30 y sus vistas W0-W30.

2. Especiales XZR, SP, PC, X29/FP y X30/LR.

3. Flags PSTATE y NZCV como resultado visible de operaciones.

4. SIMD/FP V0-V31 y vistas B, H, S, D, Q.

5. Privilegios EL0 a EL3, user mode y kernel mode.

Preguntas que responde esta unidad

Pregunta Dónde aparece
¿Qué registros puede usar un programa AArch64? Registros generales
¿Por qué existen X0 y W0? Vistas de 64 y 32 bits
¿Qué pasa cuando escribo en W0? Zero-extension hacia X0
¿Por qué SP y PC no son registros generales normales? Registros especiales
¿Qué significan N, Z, C y V? Flags
¿Qué son V0-V31? SIMD/Floating Point
¿Por qué un programa de usuario no puede tocar todo? Exception levels

Estado visible inicial

Estado Qué representa Uso en el curso
X0-X30 Registros generales de 64 bits Enteros, direcciones, argumentos, temporales
W0-W30 32 bits bajos de X0-X30 Operaciones de 32 bits
SP Stack pointer Dirección actual del stack
PC Program counter Instrucción actual o próxima según contexto
PSTATE / NZCV Estado y flags Comparaciones y saltos condicionales
V0-V31 Registros vectoriales / floating point Mapa inicial para FP y SIMD
NotaIdea clave

El procesador tiene más estado del que un programa de usuario puede tocar libremente. En esta unidad nos quedamos con el estado visible necesario para leer código y depurar programas AArch64 en Linux.

Qué no estudiaremos todavía

Tema Por qué queda fuera
ABI/AAPCS64 completo Primero hay que reconocer registros y roles básicos
Stack frames completos X29/FP y X30/LR solo serán mapa inicial
Syscalls en detalle Solo se menciona cambio de EL0 al kernel
Excepciones reales Aquí basta el mapa de niveles
MMU y páginas No son necesarias para leer registros
NEON avanzado Solo se presentan registros vectoriales
Kernel programming Esta ruta empieza desde Linux userland
